## 📊 1. 典型日志文件大小估算

### MTK Log 大小范围

| 日志类型 | 压缩大小 | 解压大小 | 文本行数 | 字符数 |
|---------|---------|---------|---------|------|
| **小型 MTK Log** | 10-30 MB | 50-150 MB | 50-200 万行 | ~1-2 亿字符 |
| **中型 MTK Log** | 50-100 MB | 200-500 MB | 200-500 万行 | ~3-10 亿字符 |
| **大型 MTK Log** | 100-300 MB | 500-1500 MB | 500-1500 万行 | ~10-30 亿字符 |

### Bug Report 大小范围

| 报告类型 | 文件数 | 总大小 | 主要内容 |
|---------|-------|-------|---------|
| **小型 Bug Report** | 10-30 个 | 50-200 MB | 少量日志 + 系统信息 |
| **中型 Bug Report** | 50-150 个 | 200-800 MB | 完整日志 + dumpsys |
| **大型 Bug Report** | 150-500 个 | 800 MB-3 GB | 完整日志 + dumpsys + tombstone + 内存导出 |

## 🔢 2. Token 转换率

### 文本到 Token 的转换 (OpenAI API)

| 语言 | 字符/Token | 单词/Token |
|------|----------|-----------|
| **英文文本** | ~4 字符 | ~0.75 单词 |
| **中文文本** | ~1-2 字符 | ~0.5 单词 |
| **日志/代码** | ~3-5 字符 | - |

### 估算公式

```
Token 估算 = (字符数 / 平均字符/Token) × (上下文膨胀系数)
```

### 上下文膨胀系数

由于需要处理、分析、总结、生成报告，实际 token 消耗通常是原始内容的:
- **最小开销**: 1.2-1.5x
- **正常模式**: 2-3x
- **深度分析**: 3-5x

## ⚡ 5. Feature Flag 优化策略

### 5.1 分析模式选择 (analysis_mode)

| 模式 | Token 节省 | 说明 |
|------|----------|------|
| **fast** | ~60-70% | 跳过深度分析、证据匹配 |
| **standard** | 100% | 平衡模式 |
| **deep** | ~150-200% | 完整分析、多轮迭代 |

### 5.2 功能开关控制

| 功能 | 启用/禁用 | 影响范围 |
|------|---------|---------|
| llm_analysis_enabled | ❌ 禁用 | 节省 ~90% Token |
| aloggrep_integration | ✅ 启用 | 预处理后输入 Token 减少 30-50% |
| knowledge_base_enabled | ✅ 启用 | 提升准确率，可能增加 10-20% Token |
| evidence_matching_enabled | ❌ 禁用 | 节省 ~20-30% Token |

## 🎯 7. 优化建议和最佳实践

### 7.1 分阶段分析策略

```
阶段 1: 快速过滤 (无 LLM，成本 $0)
  - 关键字搜索
  - 时间范围过滤
  - 进程筛选

阶段 2: 初步判断 (最少 LLM)
  - Bug 描述解析 (500 Tokens)
  - 选择是否继续深度分析

阶段 3: 深度分析 (可选)
  - 证据匹配
  - 时间线构建
  - 完整报告生成
```

### 7.2 Log 采样和截取

| 策略 | Token 节省 | 说明 |
|------|----------|------|
| 错误优先 | 30-50% | 只提交 error/fatal 级别日志 |
| 时间窗口 | 40-60% | 只分析 Bug 发生前/后 N 分钟 |
| 采样 | 50-80% | 每隔 N 行取 1 行日志 |

### 7.3 总结和摘要

| 方法 | Token 节省 |
|------|----------|
| 递归摘要 | 50-70% | 先对日志分段摘要，再对摘要分析 |
| 结构化输出 | 20-40% | 让 LLM 输出 JSON 而非自然语言 |

## 📊 8. 实际案例估算

### 案例 1: 简单 ANR 问题
- **日志规模**: 中型 MTK Log + Bug Report
- **工作流**: aloggrep + 标准分析
- **估算 Token**: ~400k-800k
- **估算成本 (GPT-4o Mini)**: $0.18-0.48
- **估算时间**: 3-5 分钟

### 案例 2: 复杂内存泄漏
- **日志规模**: 大型 MTK Log + full dumpsys
- **工作流**: QMD + evidence + deep 模式
- **估算 Token**: ~1.5M-2.5M
- **估算成本 (GPT-4o Mini)**: $0.675-1.125
- **估算时间**: 5-10 分钟

### 案例 3: 简单崩溃问题
- **日志规模**: 小型 MTK Log + tombstone
- **工作流**: fast 模式 + knowledge base
- **估算 Token**: ~80k-150k
- **估算成本 (GPT-4o Mini)**: $0.036-0.0675
- **估算时间**: 1-2 分钟

## 📝 9. 总结

### 9.1 关键指标

| 指标 | 范围 | 建议 |
|------|------|------|
| **单次分析 Token** | 80k-3M | 根据日志大小调整策略 |
